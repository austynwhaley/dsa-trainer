# DSA Trainer

A pair-programming trainer for data structures & algorithms. An AI collaborator
watches you code in real time, chimes in occasionally with inline ghost comments,
and is available for chat — but stays out of your way when you're in flow.

## Quick start

```bash
cp .env.local.example .env.local
# Add your Anthropic API key to .env.local

npm install
npm run dev
# Open http://localhost:3000
```

Python execution requires `python3` on your PATH. JavaScript runs entirely in the browser.

## Stack choices

| Concern | Choice | Why |
|---------|--------|-----|
| Framework | Next.js 14 (App Router) | API routes needed for streaming + server-side Anthropic calls. File-based routing saves setup. |
| Editor | Monaco Editor | VS Code's editor. Best JS/Python language support out of the box, stable inline decoration API for ghost comments. |
| Styling | Tailwind CSS v4 | Fast iteration, no CSS files to manage. |
| State | Zustand | Lightweight global store. No boilerplate, no context providers. |
| LLM | Anthropic SDK (claude-sonnet-4-6) | Full streaming, best code reasoning. |
| Ghost comments | claude-haiku-4-5 | Haiku is fast/cheap for the high-frequency ghost endpoint. |
| JS execution | Web Worker (in-browser) | No server needed, zero latency, fully isolated. |
| Python execution | `python3` subprocess (server-side) | Simpler than Pyodide for v1, no 50MB WASM download. |
| Deploy | Vercel | Zero-config with Next.js, handles streaming natively. |
| Database | None | Single-player, local-first. No persistence needed in v1. |

## Architecture

```
app/
  page.tsx              — Three-panel layout (problem | editor | chat)
  api/
    chat/               — Streaming chat with Anthropic
    ghost/              — Ghost comment generation (debounced, Haiku)
    generate-problem/   — AI problem generation
    run-code/           — Python execution (JS runs client-side)

components/
  CodeEditor.tsx        — Monaco wrapper, ghost overlay, run button
  ChatPanel.tsx         — Streaming chat UI
  ProblemPanel.tsx      — Problem description + problem switcher

lib/
  store.ts              — Zustand global state
  types.ts              — Shared types
  problems.ts           — 10 hardcoded problems
  context.ts            — Builds structured AI context from session state
  system-prompt.ts      — Loads system.md and injects context

prompts/
  system.md             — THE AI PERSONALITY. Edit this to tune behavior.

public/workers/
  js-runner.js          — JavaScript Web Worker for in-browser execution
```

## The system prompt

`prompts/system.md` is the AI's personality. It's loaded at runtime (not compiled in),
so you can edit it and see changes on the next API call without restarting.

Key design decisions in the prompt:
- **Peer, not teacher.** The AI doesn't explain things unprompted.
- **Opinionated.** It says when an approach is bad.
- **Concise by default.** 1-3 sentences unless asked for more.
- **Ghost mode.** When generating inline comments, it responds with ONLY the comment
  text — no preamble, under 80 chars.

To tune the personality, edit `prompts/system.md` directly.

## Adding problems

Problems live in `lib/problems.ts` as a `Problem[]`. Each problem has:

```typescript
{
  id: "slug",
  title: "Problem Title",
  difficulty: "easy" | "medium" | "hard",
  topic: "arrays",
  description: "markdown string",
  starterCode: { javascript: "...", python: "..." },
  testCases: [{ input: "...", expectedOutput: "..." }],
}
```

Test case inputs are strings that get `JSON.parse`d and spread as function arguments.
So `input: "[1,2,3], 9"` → `fn([1,2,3], 9)`.

You can also generate problems with AI (the "Change" tab in the problem panel)
or import arbitrary problem JSON.

## Ghost comments

Ghost comments appear as greyed-out inline annotations after ~4 seconds of
typing silence. They're generated by Haiku (fast, cheap) with a 15-second
cooldown so they feel rare and earned.

- They disappear when you start typing again.
- Press Escape to dismiss them manually.
- They only trigger if your code is > 20 chars (no ghosts on empty buffers).
- The cooldown means you won't see more than ~4 per minute.

## Known limitations

- JavaScript test runner uses a function-name allowlist (hardcoded problem names).
  Custom/imported problems with different function names may not auto-detect. 
  Fix: add the function name to the switch in `public/workers/js-runner.js`.
- Python runner uses `python3` subprocess — no sandboxing beyond a 10s timeout.
  Don't run untrusted code.
- LeetCode import isn't implemented — the "Import" tab accepts raw JSON in the
  Problem shape. You can paste problem text into chat and ask the AI to format it.
- No session persistence. Refreshing clears everything.
- Single language at a time. Switching language resets the code buffer.
